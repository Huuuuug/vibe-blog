# Codex 协作规范

这份文档用于约束 Codex 在本仓库中的代码产出方式，目标是让新增代码在风格、结构、可维护性和验证方式上保持稳定。

适用范围：

- 新功能开发
- 缺陷修复
- 重构
- 文档补充
- Code review / 自检

## 1. 协作目标

Codex 在本仓库中的默认目标不是“尽快写出能跑的代码”，而是：

1. 优先和现有项目结构保持一致。
2. 优先产出小步、可审查、可验证的改动。
3. 优先类型安全和可维护性，不做无必要的炫技实现。
4. 优先遵守仓库已有规则，再提出额外优化建议。

## 2. 仓库当前事实

当前仓库已经明确的技术约束：

- 框架：`Next.js 16.2.1`
- React：`19.2.4`
- 语言：`TypeScript`
- 样式：`Tailwind CSS 4`
- Lint：`ESLint 9 + eslint-config-next`
- 路径别名：`@/* -> src/*`
- TypeScript：`strict: true`

Codex 必须注意：

- 这是一个和旧认知可能不完全一致的 `Next.js` 版本。
- 修改 `Next.js` 相关代码前，应先阅读 `node_modules/next/dist/docs/` 中相关文档。
- 不要凭旧版本经验直接假设 API、目录结构或约定仍然成立。

## 3. 默认工作方式

除非我明确说明，否则 Codex 应默认按以下方式执行：

1. 先阅读相关文件和上下文，再动手修改。
2. 先理解现有实现模式，再补代码，不凭空发明新结构。
3. 以最小必要改动完成目标，不顺手大改无关代码。
4. 改完后必须进行自检，并尽量执行可用验证命令。
5. 如果发现存在多种实现方式，优先选择和现有代码最一致的那一种。

## 4. 代码风格要求

### 4.1 通用要求

- 保持代码简洁，避免过度抽象。
- 单个函数只做一类事情，职责要清楚。
- 命名必须直接表达语义，避免缩写和模糊命名。
- 非必要不增加新依赖。
- 非必要不引入新的全局工具函数、hooks、配置层。
- 非必要不写“为了以后可能会用”的代码。

### 4.2 TypeScript 要求

- 不要绕过 `strict` 模式。
- 非必要禁止使用 `any`。
- 能用明确类型时，不要依赖模糊推断。
- 对外暴露的数据结构要有清晰类型定义。
- 空值、可选值、边界值要显式处理。

### 4.3 React / Next.js 要求

- 优先遵循 App Router 现有模式。
- 区分 Server Component 和 Client Component，避免无意义地加 `"use client"`。
- 数据获取逻辑不要随意下沉到客户端。
- 页面代码负责组合，复杂逻辑尽量沉到合适的 `lib` 或独立组件。
- 涉及路由、缓存、metadata、SEO 时，先确认当前 Next.js 16 的官方约定。

### 4.4 样式要求

- 优先复用现有设计语言和组件模式。
- 不要为了小改动引入全新的视觉体系。
- Tailwind class 保持可读，不堆砌难维护的超长串。
- 样式修改应兼顾桌面端和移动端。

## 5. 目录与职责约束

默认职责边界如下：

- `src/app`：路由页面、布局、路由级元数据、路由处理器
- `src/components/ui`：纯公共组件，业务无关，可跨页面直接复用
- `src/components/shared`：站内通用展示组件，承载轻量语义但不绑定业务实体
- `src/components/layout`：站点布局与导航骨架组件
- `src/components/features`：按业务域组织的组件，不作为基础公共组件滥用
- `src/lib`：业务逻辑、数据访问、映射、工具函数、配置
- `docs`：架构说明、协作规范、领域文档

约束：

- 页面层不要直接堆积太多数据转换逻辑。
- UI 组件不要直接依赖底层第三方响应结构。
- 第三方数据源应先经过 `mapper` 或封装层，再进入 UI。
- 工具函数放入 `lib` 前，先确认是否真的会复用。

## 6. 变更边界

除非我明确要求，否则默认禁止：

- 大范围重构
- 批量重命名
- 修改公共接口但不处理调用方
- 为了“更优雅”而改动稳定逻辑
- 混入与当前需求无关的修复
- 擅自新增依赖或升级核心版本

如果确实需要超出边界，Codex 应先说明原因，再执行。

## 7. 完成定义

一个任务只有在满足以下条件时才算完成：

1. 代码实现满足需求。
2. 改动范围受控，没有引入明显无关修改。
3. 类型、空值、异常路径经过基本处理。
4. 关键逻辑具备可读性，不需要靠大量口头解释才能理解。
5. 已执行可行的验证步骤，或明确说明无法验证的原因。

## 8. 验证要求

当前仓库已知可用命令：

```bash
pnpm lint
```

推荐 Codex 在条件允许时额外检查：

```bash
pnpm exec tsc --noEmit
pnpm build
```

说明：

- 如果任务只改文档，可不跑构建。
- 如果任务改动了页面、组件、数据映射或路由，至少应优先尝试 `lint`。
- 如果无法执行验证，必须明确说明原因，而不是默认跳过。

## 9. 自检清单

Codex 在提交结果前，应至少检查这些问题：

- 是否引入了不必要的复杂度？
- 是否和仓库已有写法明显不一致？
- 是否存在空值、未定义、异常分支遗漏？
- 是否有服务端 / 客户端边界使用错误？
- 是否破坏已有 SEO、缓存、路由或数据映射逻辑？
- 是否出现了“能跑，但后续难维护”的写法？

如果用户要求 review，优先输出：

1. 明确的问题
2. 风险和回归点
3. 测试缺口
4. 最后才是总结

## 10. 给 Codex 的固定指令模板

以后可以直接把下面这段贴给 Codex：

```text
请按仓库协作规范执行本次修改：

1. 先阅读相关文件和必要上下文，再开始改动。
2. 遵守现有项目结构和命名方式，优先最小改动。
3. 这是 Next.js 16 项目，涉及框架代码前先核对相关官方文档，不要沿用旧版本习惯。
4. 保持 TypeScript strict 兼容，非必要不要使用 any。
5. 不要顺手重构无关代码，不要新增不必要依赖。
6. 改完后做自检，并执行能执行的验证命令。
7. 输出结果时请说明：改了什么、为什么这样改、如何验证、还有什么风险。
```

## 11. 更严格模式

如果你希望 Codex 更保守，可以再附加这些要求：

```text
额外要求：

- 先列出将要修改的文件，再开始编辑。
- 每次只解决当前任务，不处理附带问题。
- 如果需要超过 3 个文件的改动，先说明原因。
- 如果需要改公共接口，先说明影响范围。
- 如果无法确认某个 Next.js 16 用法，先查文档再写代码。
- 没有验证就不要默认说“完成”。
```

## 12. 推荐后续增强

如果你想继续提高代码质量，建议后续补齐这些工程约束：

1. 在 `package.json` 增加 `typecheck` 脚本。
2. 增加统一格式化工具，例如 `Prettier`。
3. 为关键数据映射和工具函数补最小测试。
4. 补一份更明确的架构文档，定义页面层、数据层、组件层边界。
5. 对常见改动场景沉淀示例，例如“新增页面”“新增 Notion 字段”“修改列表卡片”。
